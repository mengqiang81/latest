* 注意，领域是一等公民，所以聚合根命名上不要加Entity之类的后缀，领域服务也不是非得用接口，接口是为了依赖倒置的，一等公民本来就是被别人依赖的
* 要注意区别额外规则和领域事实，比如可能有一些规则不允许产生负余额，但是不能改变余额为负的事实，所以一个合法的余额领域模型是可以为负的
* 不使用exception来实现应急管理
* 对于需要放入flink中的算子，属于业务领域规则，包括数据筛选规则和计算规则等，需要在领域层定义，可以被序列化传输到大数据引擎去执行。简化的架构是基础设施层直接耦合大数据引擎，根据业务规则构建flinksql等，也是没有问题的。
* 对于存在跨领域算子的情况，还是得把各领域的UDF放到一个统一的数据协调器上计算，因为跨领域rpc调用的开销无法接受。一定要保证领域层足够纯净，否则将很难打包进UDF，基础设施层要为大数据计算做一个独立模块。
* * 因为CQRS和DDD是两种架构，可以结合，但还是要注意区分，所以domain里不要出现CQRS的概念
